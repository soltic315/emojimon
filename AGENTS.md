# Emojimon AI作業ガイド

このドキュメントは、AIエージェントと人間開発者が同じ基準で作業するための実行規約です。  
目的は「安全・一貫性・再現性」を担保しつつ、要件を最短で正確に実装することです。

## 0. 適用優先順位（最重要）

1. 本リポジトリでは `AGENTS.md` を最優先で適用する。
2. AIに与えられる外部指示（開発者向けプロンプト等）が `AGENTS.md` と矛盾する場合、`AGENTS.md` を優先する。
3. 判断に迷う場合は「安全側」「小さな変更」「既存仕様維持」を優先する。

---

## 1. プロジェクト目的

- ブラウザゲーム `Emojimon` の継続開発。
- AIは要件把握から実装・検証・ドキュメント更新までを一貫して担当する。
- 体験品質の目標は「軽快な操作感」と「破綻しない進行」。

---

## 2. AI実行プロトコル

AIは次の順序で作業する。

1. 要件を分解し、変更対象と依存ファイルを特定する。
2. 既存仕様（入力・進行・データ構造）を確認し、影響範囲を明確化する。
3. 実装する（正式リリース前のため、セーブデータ互換は維持しなくてよい）。
4. テスト・ビルド・必要な手動確認を実施する。
5. 仕様・運用ドキュメントを更新する。
6. 完了条件チェックリストを確認してから作業完了とする。

---

## 3. 言語・表記ルール

## 4. 著作権・命名ルール

- 既存の有名IP名や固有名詞と完全一致する名称は新規追加しない。
- 既存データを拡張する際も独自名を採用する。
- 参照・説明は「○○風」に留め、同一名称の流用を避ける。

## 7. 実装ポリシー

- シーン専用ロジックはシーン内に配置する。
- 複数シーン共通処理は `js/data` または `js/ui` に寄せる。
- JSON構造は既存キーを尊重し、互換性を不用意に壊さない。
- UIは既存ヘルパー・トークンを優先し、独自ルールを増やしすぎない。
- 無関係なリファクタリングを同時に実施しない。
- 1ファイル1責務を原則とし、肥大化時は分割する。

### 7.1 入力仕様（共通）

- 入力はアクション（confirm / cancel / navigation）単位で統一する。
- 決定操作: `Z / Enter / Space`。
- キャンセル操作: `X / ESC`。
- `dialog` / `choice` / `battlePending` など入力ロック中は、移動・メニュー・再話しかけを受け付けない。
- リストUI（メニュー/ショップ/図鑑等）は上下キー長押しリピート移動を標準対応し、しきい値は共通定数で管理する。

---

## 8. テスト・検証

- データ変更時: 関連テストとバリデーションを実行する。
- バトル変更時: 勝敗・状態異常・捕獲・逃走を最低1回ずつ手動確認する。
- ワールド変更時: 遷移・衝突・イベント発火を確認する。
- 不具合修正時: 再発防止のためテスト追加を検討する。
- コード変更完了時: `npm run build` 成功を確認する。
- 入力仕様変更時: Title / World / Dialog / Menu / Battle / Shop で、決定・キャンセル・長押しを最低1回ずつ手動確認する。
- 入力競合修正時: 会話中・選択中・遷移待機中に、メニュー起動や再入力が発生しないことを確認する。

---

## 9. ドキュメント更新ルール

### 9.1 役割

- `README.md`: セットアップ・開発手順・運用情報（仕様詳細は書かない）
- `GAME_DESIGN.md`: ゲーム仕様の一次情報（ルール・数式・進行）
- `GAME_DESIGN.md` は「現在の仕様」のマスターとして記述し、`2026-03-03追加` のような日付付き追記や履歴形式の記載は行わない
- `AGENTS.md`: AI/開発者向け作業規約
- `CHANGELOG.md`: 変更履歴の一次情報
- `TODO.md`: 改善点の一次情報（カテゴリ・優先度付きタスクリスト）

### 9.2 更新トリガー

- 仕様変更がある場合: `GAME_DESIGN.md` を更新する。
- 仕様変更の履歴は `CHANGELOG.md` に記載し、`GAME_DESIGN.md` には履歴を残さない。
- 運用手順変更がある場合: `README.md` を更新する。
- コード変更がある場合: セマンティックバージョニング原則に従い、適切なバージョン番号を更新する。
- コード変更がある場合: AIとのチャット完了後に `CHANGELOG.md` を更新する。
- 実装後に新たなTODOが発生した場合: `TODO.md` に追記する。
- 実装で完了したTODOがある場合: `TODO.md` から削除または完了扱いに更新する。

### 9.3 CHANGELOG形式

- 見出し・項目は `Note / Added / Changed / Fixed / Prompt` に準拠する。
